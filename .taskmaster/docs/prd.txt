Overview

CatSafe Plants is a web app that lets cat owners quickly check whether a plant or flower is safe, mildly toxic, highly toxic, or unknown for cats—using fast search + a dedicated plant page with citations. If a plant is toxic, the product shows 3–5 safe alternatives (that link to real plant pages) so users can choose a safer option immediately. An admin-only workflow allows generating AI-suggested alternatives and approving them before anything is shown publicly.

Core Features
1) Plant Search (common/scientific/aka) with autocomplete

What it does: Users search plants by common name, scientific name, or aka names; results show matches with a safety badge.

Why it’s important: The main job-to-be-done is “tell me fast if this plant is safe.” Search speed and match quality drive trust and usability.

How it works (high level):

Index plant names (common/scientific/aka).

Search supports prefix + contains matching.

Autocomplete suggests likely matches; selecting opens the plant page.

2) Plant Detail Pages (status + toxicity info + citations)

What it does: A plant page shows photo (URL), common/scientific name, safety status, toxicity symptoms (if toxic), toxic parts (if toxic), severity (derived), and citations (2–3 links).

Why it’s important: This is the “trust page” where the user decides what to do. Citations are mandatory for credibility.

How it works (high level):

Render fields from plants + citations.

Safety status drives conditional UI (toxic vs unknown vs safe).

Severity is derived from safety_status.

3) Safe Alternatives for Toxic Plants (3–5, clickable)

What it does: If a plant is toxic, show 3–5 non-toxic alternatives that link to their own plant pages.

Why it’s important: It turns “bad news” into a solution: “buy this instead.”

How it works (high level):

Use plant_alternatives join table linking toxic plant => safe plant.

Only show approved alternatives (via admin workflow).

Enforce that alternatives resolve to real plant pages.

4) Unknown Status + Caution Note

What it does: If a plant is unknown, show “Unknown” and a caution note.

Why it’s important: Avoids false certainty; preserves trust and safety.

How it works (high level):

safety_status = unknown triggers the caution UI and suppresses alternative suggestions unless curated.

5) Admin Access (allowlist email)

What it does: /admin is accessible only if the user’s email is allowlisted (robertxluo@gmail.com
).

Why it’s important: Prevents public edits; keeps data quality high.

How it works (high level):

Auth gate on admin routes.

Server-side check against allowlist.

6) Admin CRUD for Plants + Citations

What it does: Admin can add/edit plant entries and manage citations.

Why it’s important: Data quality and citation accuracy determine product credibility.

How it works (high level):

Admin UI forms for plant fields + citations list.

Validation for required fields and URL formatting.

7) Admin-only AI Suggestions for Alternatives (approve/reject)

What it does: Admin can request AI suggestions for safe alternatives for a toxic plant, then approve/reject before public display.

Why it’s important: Speeds curation while keeping human review as the quality gate.

How it works (high level):

Input: toxic plant id.

AI proposes 3–5 non-toxic alternatives from DB (or candidates to add).

Suggestions are queued; nothing publishes until approved.

User Experience
User personas

Primary: Cat owner who buys indoor plants/flowers and needs a fast, trustworthy safety check.

Secondary (Admin): Site curator managing plant data, citations, and alternative recommendations.

Key user flows

Home => Search => Plant Page

User lands on home, types into search, sees autocomplete, selects a plant, views plant detail.

Toxic Plant => Alternatives

User opens toxic plant page, reads symptoms/toxic parts, clicks a safe alternative to view its page.

Unknown Plant

User opens plant page with “Unknown,” sees caution note, uses search again.

Admin: Review + Generate AI + Approve

Admin enters /admin, sees toxic plants missing alternatives and/or suggestion queue, generates AI suggestions, approves selected suggestions, sees them appear on public plant pages.

UI/UX considerations

Minimal, clean layout; consistent badges for safety status.

Safety status is visually prominent (badge + short label).

Toxicity fields are conditional (only show symptoms/toxic parts when toxic).

Citations are clearly separated and easy to scan (2–3 authoritative links).

Alternatives are presented as a short list/cards with clear click targets.

Admin screens optimize for throughput: “missing alternatives” list + “suggestions queue.”

Risks and Mitigations
Data trust risk (incorrect safety info / weak citations)

Mitigation: Require citations for seeded plants; display citations prominently; keep unknown status + caution note; admin-only edits.

Search quality risk (users can’t find plants via expected names)

Mitigation: Support aka names + contains matching; test with a real list of common queries; add relevance ranking later without changing the core contract.

AI suggestions risk (hallucinated or inappropriate alternatives)

Mitigation: Admin-only + approval gate; restrict AI to selecting from existing non_toxic plants when possible; store rationale for auditability.

MVP scope creep risk

Mitigation: Enforce non-goals: no accounts, no favorites, no photo uploads, no plant ID-by-photo, no mobile app.

Resource constraints (time/attention)

Mitigation: Prioritize “public search + plant page + citations” first; ship alternatives manually before AI; keep admin UI minimal.
